# Persona-Based Service Testing

고객 페르소나를 정의하고 실제 서비스를 **고객처럼 사용**하여 사용성, PMF(Product-Market Fit), 업무 워크플로우 적합성을 검증하는 스킬.

**핵심 원칙**: 이 스킬은 E2E 테스트가 아니다. "기능이 동작하는가?"가 아니라 **"이 서비스로 실제 업무를 할 수 있는가?"**를 검증한다.

## Phase 1: 준비

### 1-1. 테스트 대상 인터페이스 선택

사용자에게 테스트할 인터페이스를 확인한다. 복수 선택 가능.

| 인터페이스 | 설명                   | 도구           |
| ---------- | ---------------------- | -------------- |
| `web`      | 브라우저 기반 웹 UI    | Playwright MCP |
| `cli`      | CLI 명령어             | Bash tool      |
| `mcp`      | MCP 도구 직접 호출     | MCP tools      |
| `api`      | REST/GraphQL API       | Bash (curl)    |
| `plugin`   | hook/plugin 워크플로우 | Edit + Bash    |

- 복수 선택 시 각 인터페이스를 **독립적으로** 테스트
- `--cross` 옵션 추가 시 인터페이스 간 **크로스 시나리오**도 실행

### 1-2. 프로젝트 설정 로딩

**CLAUDE.md에서 `Persona Test` 섹션을 찾아** 프로젝트별 설정을 로딩한다.

찾아야 할 정보:
- **인터페이스 URL/명령어**: 각 인터페이스의 접속 주소 또는 실행 명령
- **테스트 계정**: 계정 정보 파일 위치 (예: `docs/test-accounts.md`)
- **제품 스펙**: 제품이 해결하는 문제와 핵심 가치 문서
- **프로젝트 페르소나**: 미리 정의된 페르소나 (있으면 사용, 없으면 Phase 2에서 생성)

CLAUDE.md에 설정이 없으면 사용자에게 직접 확인:
1. 서비스에 어떤 인터페이스가 있는가?
2. 테스트 계정은 어디에 있는가?
3. 주요 사용자 역할은 무엇인가?

> 프로젝트 관리자를 위한 CLAUDE.md 설정 가이드: `references/claude-md-setup.md` 참조

### 1-3. 사전 확인

선택한 인터페이스별로 서비스가 접근 가능한지 확인한다.

## Phase 2: 페르소나 정의

### 좋은 페르소나의 조건

- 서비스의 **다른 측면**을 사용하는 역할 (예: 관리자 vs 실무자)
- **구체적인 업무 목표**를 갖고 있음 (예: "오늘 해야 할 3개 작업을 파악하고 하나를 완료 처리")
- 각 페르소나마다 고유한 **pain point 가설**이 있음
- "오늘의 시나리오"는 5-10분 분량의 현실적 업무 흐름

### 페르소나 템플릿

| ID  | 이름 | 역할 | 업무 목표 | 오늘의 시나리오 | 핵심 검증 |
| --- | ---- | ---- | --------- | --------------- | --------- |

CLAUDE.md에 프로젝트 페르소나가 정의되어 있으면 사용한다. 없으면 제품 스펙과 사용자 입력을 기반으로 3-5개 페르소나를 제안한다.

> 인터페이스별 페르소나 예시: `references/persona-templates.md` 참조

## Phase 3: 시나리오 기반 시뮬레이션

각 페르소나가 **실제 업무 상황에서 하는 행동**을 순서대로 수행한다.

### 시나리오 설계 원칙

- 출발점은 해당 인터페이스의 **가장 자연스러운 진입점**
- "기능이 동작하는가?"가 아니라 **"이 사람이 이 도구로 자기 업무를 자연스럽게 해결할 수 있는가?"**
- 각 시나리오는 **5-10분 분량**의 현실적 업무 흐름

### 인터페이스별 테스트 가이드

**Web UI**: Playwright MCP로 브라우저 조작. 스크린샷은 `persona-test/` 디렉토리에 저장.

**CLI**: Bash tool로 명령어 실행. 명령어 + 출력을 마크다운 코드블록으로 기록.

**MCP**: MCP tools 직접 호출. 요청/응답을 기록.

**API**: curl 또는 HTTP 클라이언트로 호출. 요청/응답을 기록.

**Plugin**: Edit + Bash로 hook 트리거 관찰. hook 메시지 + 트리거 상황을 기록.

### 시뮬레이션 중 관찰 포인트

- **마찰 지점**: 원하는 동작을 하는데 예상보다 많은 단계가 필요한 곳
- **혼란 지점**: 다음에 무엇을 해야 할지 모르는 순간
- **실망 지점**: 기대한 기능이 없거나 다르게 동작하는 순간
- **감동 지점**: 예상보다 편리하거나 잘 설계된 부분
- **빈 상태 대응**: 데이터가 없을 때 사용자가 길을 잃지 않는가
- **정보 계층**: 중요한 정보가 눈에 잘 띄는가

### 크로스 인터페이스 (`--cross` 옵션 시)

독립 시나리오 완료 후, 인터페이스 간 데이터 동기화와 일관성을 검증한다:
- 한 인터페이스에서 변경한 내용이 다른 인터페이스에 즉시 반영되는가?
- 동일 데이터의 표현이 인터페이스 간 일관적인가?

## Phase 4: 발견사항 분류

### 분류 체계

- **Bug**: 기능이 스펙과 다르게 동작 (Critical/High/Medium/Low)
- **Friction**: 되긴 되는데 불편하거나 비직관적
- **Gap**: 업무에 필요한데 없는 기능 — PMF 관점에서 치명적인 부재
- **Delight**: 잘 된 부분 — 유지하거나 강화해야 할 것

### 인터페이스별 관찰 포인트

**Web UI**: 클릭/단계 수, 시각적 계층, 반응 속도, 빈 상태 UI, 에러 상태 UI, 단축키, 접근성

**CLI**: 에러 메시지→다음 행동 안내 품질, 출력 가독성, `--output json/table` 전환, 파이프 친화성, help 품질, 명령어 직관성

**MCP**: AI 에이전트가 응답만으로 다음 행동 결정 가능성, 에러 응답 구조화, 빈 상태 응답 유용성, 토큰 효율성

**API**: 응답 구조 일관성, 에러 코드/메시지 품질, 페이지네이션, 버전 관리

**Plugin**: hook 방해도, 메시지 명확성, 등록 후 투명한 동작

### 출력 양식

```markdown
## PMF 체크리스트
(테스트한 인터페이스별로 섹션 생성)

### {인터페이스명}
- [ ] 핵심 가치 전달: 첫 사용 30초 안에 서비스의 가치를 느낄 수 있는가?
- [ ] 핵심 워크플로우: 페르소나의 일상 업무를 서비스 안에서 완료할 수 있는가?
- [ ] 차별화: 기존 도구 대비 명확한 차이를 체감할 수 있는가?

## 버그
| #   | 심각도 | 인터페이스 | 설명 | 영향 페르소나 |
| --- | ------ | ---------- | ---- | ------------- |

## 마찰 지점 (Friction)
| #   | 인터페이스 | 설명 | 현재 동작 | 기대 동작 | 영향 페르소나 |
| --- | ---------- | ---- | --------- | --------- | ------------- |

## 기능 부재 (Gap)
| #   | 인터페이스 | 설명 | 업무 시나리오 | 현재 대안 | PMF 영향도 |
| --- | ---------- | ---- | ------------- | --------- | ---------- |

## 잘 된 부분 (Delight)
| #   | 인터페이스 | 설명 | 영향 페르소나 |
| --- | ---------- | ---- | ------------- |

## 종합 우선순위 (Top 5)
1. ...
```

## Phase 5: 근본 원인 분석 (선택)

사용자가 요청하면 발견된 문제의 근본 원인을 프로젝트 코드에서 추적하고 수정한다.

## Tips

- **고객처럼 생각하기**: "이 버튼이 동작하나?"가 아니라 "내가 이 역할이라면 뭘 하고 싶을까?"
- **비교 관점**: 같은 작업을 다른 인터페이스로 해보면 일관성 문제 발견
- **첫인상 기록**: 처음 보는 출력/화면에서 가장 먼저 눈에 들어오는 것 기록
- **5초 테스트**: 출력을 5초 보고 "오늘 내가 해야 할 일이 뭔지 알 수 있는가?"
- **문제 중심 기록**: 문제 발견 시에만 상세 기록 (문제 없으면 스킵)
